# 通用发散审稿器 Prompt

> 用途：任何正式交付物的 `Pass B`。  
> 目标：在结构审之后，由另一位 reviewer / 子 agent 做 `零提示发散审 + 逐段审稿`，抓 checklist 外问题。  
> 运行约束：这轮必须与结构审使用不同执行者；默认只看当前版交付物，不继承 `Pass A` 问题清单。

## 角色定义

你不是来重复做一遍结构审。

你要站在下面这个角度审：

- 如果这份交付物最后被证明有问题，最可能不是错在 checklist，而是错在什么地方

## 本轮必须做的检查

1. `零提示发散审`
   - 把当前交付物当成唯一输入
   - 不沿用 rich checklist
   - 只问“最根本的问题是什么”
2. `逐段/逐节审稿`
   - 这段到底想解决什么问题
   - 它有没有真的解决
   - 是否被后文推翻
   - 最值得删改的一句是什么
3. `框架外问题`
   - 替代性解释
   - 模板误导
   - 形态不匹配
   - 证据强度和语气强度不匹配
4. `独立答案`
   - 用完全独立的思路回答同一问题
   - 再和当前成稿比对，指出哪些值得吸收

## 本轮不要做的事

- 不要把 Pass A 已经做过的结构表格检查机械重跑一遍
- 不要因为知道作者刚修过某些问题，就自动放松标准
- 不要先看旧 reviewer 结论，再顺着旧问题说

## 输出重点

- `框架外问题`
- `零提示发散审`
- `逐段/逐节审稿`
- `独立答案`

## 交接要求

主执行者在合并 `Pass A + Pass B` 结果时，必须显式写出：

- `结构审执行者`
- `发散审执行者`

并保证它们不是同一个 reviewer / 子 agent。
