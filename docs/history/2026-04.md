## 2026-04-24

### 首页关键情报解读层收口

- `editor_payload` 新增共享情报解读后处理：
  - 原始外部情报、结构化催化证据、`event_digest` lead、`briefing` 新闻行都会补齐 `信号类型 / 信号强弱 / 主要影响 / 结论 / 传导`
  - 传导表达统一写成“先影响什么，再看什么验证”，避免首页关键新闻只剩标题、来源和标签
  - 已补 `AI硬件 / AI应用 / 创新药 / 电网特高压 / 政策 / 财报日历 / 地缘 / 利率 / 黄金原油 / 资金承接 / 卖方共识` 等常见信号解释
- 同时修掉一个相关性副作用：
  - 情报相关性打分现在只看新闻标题/原始证据主体，不再因为后处理模板里出现 `订单 / 主题预期 / 标的行业` 等词就把无关新闻误放进首页

  - `南方电网` 这类无关电网新闻不会再被 AI 基金的主题兜底污染成相关情报
- 真实样例复核时顺手修掉 `Tushare dividend` 单位解释：
  - `cash_div / cash_div_tax / stk_bo_rate / stk_co_rate / stk_div` 是每股口径，展示成 A 股常用“每10股”时必须乘以 10
  - 新易盛 2025 年报分红不再被误写成 `每10股派现 1.00 元`，而是 `每10股派现 10.00 元，每10股转增 4.00 股`
- 个股聚合层补上分数-结论硬 gate：
  - `technical < 30 / catalyst < 20 / risk < 20` 任一项未过时，`cn_stock` 结论不得再输出 `较强机会 / 强机会`
  - 两项及以上未过直接压回 `无信号`；单项未过封顶 `有信号但不充分`
  - 同步下沉到 `trade_state / action_plan / discover_next_step`，避免 stale rating 或下一步命令把坏信号继续往后推
  - `release_check / report_guard` 新增 `L040` 与共享 `score_consistency`，发布前会拦住“低技术/低催化/低风险特征却给强机会”的稿件
  - 真实 spot check：`python -m src.commands.stock_analysis 300502 --today` 现在首屏为 `— 无信号`，并明确写出 `技术面27/30、催化面18/20、风险特征11/20` 的硬门槛原因

## 2026-04-23

### Horizon 合同升级为 rich code + family_code

- 共享 `horizon` 合同这轮不再只保留 `watch / short_term / swing / position_trade / long_term_allocation` 这几个粗桶：
  - `code` 现在升级成细粒度 rich code，例如 `watch_crowded_mainline_consolidation`、`swing_mainline_rotation_followthrough`、`position_trade_thesis_stated`
  - 同时补了稳定兼容层 `family_code`，供 `trade_handoff / theme_playbook / client_report / etf_pick track bucket / portfolio / review` 继续按粗周期分桶
  - 合同里还新增了 `setup_code / setup_label`，为后续把表达层交给 LLM 或 editor 生成做准备
- 这轮重点不是“多几个枚举名”，而是把 `高位拥挤主线分歧` 正式提升成共享状态：
  - `build_analysis_horizon_profile(...)` 现在会同时吃 `price_percentile_1y / RSI / 假突破 / 情绪指数 / 压力位邻近 / phase_label`
  - 对高位主线里的拥挤分歧，不再因为 `technical < 35` 就粗暴落成 `修复早期`
  - 今天这类 `CPO / 通信设备 ETF` 的观察稿，表达重点会转成“高位主线分歧消化 / 强势整理”，而不是“底部修复”
- 兼容面也一起补齐：
  - `trade_handoff` 现在按 `family_code` 决定时段说明和组合预演口径，不会被 rich code 打断
  - `client_report / theme_playbook / etf_pick` 的轨道分桶也改吃 `family_code`
  - 旧快照里如果还是老的粗 code，兼容 helper 仍能正常识别

### ETF 主题票排序与观察 gate 收口

- `etf_pick` 这轮把“识别对了主题但 winner 还是旧对象”的链路断点收掉了：
  - finalists 在 `hydrate full profile` 后会立刻按最新 `metadata / rating / action` 重新排序
  - `client_payload.catalyst_analyses` 也改为复用这批已 hydration 的 finalists，不再把 pre-hydration 的旧 `coverage_analyses` 原样塞进成稿 sidecar
- ETF 排序现在额外吃一层 `主题纯度 / 指数显式度`：
  - dedicated carrier（指数名/基金名就直写 `通信设备 / CPO / 创新药 / 游戏` 等）会高于只靠持仓反推主链的 broad wrapper
  - 目标不是硬改结果，而是让“更纯、更直连的载体”在同链路竞争里拿到合理优先级
- `opportunity_engine` 新增 fund-like shared gate：
  - 对 `theme_directness in {direct, adjacent, application, broad, sidechain, non_ai, geopolitical}` 的 ETF / 基金 / 指数产品
  - 只要 `技术 < 50` 或 `催化 < 20`
  - 评级直接压回观察级，`trade_state / action / discover_ready_for_next_step` 同步承认这条 gate
- 这次修的是共享聚合层，不是 renderer hack：
  - rating
  - action
  - pick readiness
  - ETF final sorting
  现在都吃同一套“主题票先过确认，再谈推荐”的合同
