# Markdown 审稿闭环

## 什么时候触发

只要当前任务会输出一份可独立阅读的 `Markdown` 报告，就默认触发这套闭环。典型包括：

- `stock_pick`
- `briefing`
- `ETF 分析`
- `场外基金分析`
- 其它研究型成稿

这里的关键判断标准不是“用了哪个命令”，而是“有没有交付一份 Markdown 成稿”。

---

## 默认流程

1. 先生成第一版 `Markdown`
2. 调用外部金融专家审稿器做独立审稿
3. 根据反馈修正代码、数据口径、展示层或交易参数
4. 重新生成新版 `Markdown`
5. 再次审稿
6. 直到外部审稿明确表示没有新的实质问题

不要把第一版成稿直接交付给用户。

---

## 外部审稿器怎么理解

外部审稿器是“第二视角”，默认要做到：

- 脱离项目代码上下文
- 联网核验价格、估值、财报日、新闻、监管、交易参数
- 做最小必要的回测或历史复盘
- 给出明确的反馈意见，而不是泛泛而谈

实际 prompt 存在：

- [`docs/prompts/external_financial_reviewer.md`](../../docs/prompts/external_financial_reviewer.md)

主执行者的闭环 prompt 存在：

## 哪些任务可以并发

如果用户一次明确要多份互相独立的报告，例如：

- `stock_pick`
- `briefing`
- `ETF 分析`
- `场外基金分析`

可以并发开始生成。  
但要注意两点：

1. 每份报告各自独立产出，不共用“结论稿”
2. 每份报告都要各自完成一轮外部审稿闭环，不能只审其中一份

也就是说，可以并发“开始”，不能并发后“合并糊成一个没审完的交付物”。

---

## 什么时候停止循环

只有在外部审稿明确表示：

- 没有新的 `P0 / P1` 问题
- 剩余主要是展示优化

才可以停止。

如果还有以下问题，就继续循环：

- 事实错误
- 口径混用
- 逻辑断裂
- 展示与评分不一致
- 交易参数硬伤

---

## 这套规则的目的

这套规则不是为了把流程变复杂，而是为了避免两种常见错误：

1. 把“项目内部自洽”误当成“对外可信”
2. 把“第一版 Markdown”误当成“可交付终稿”

它的本质是把工作流固定成：

`生成 -> 外部审稿 -> 修正 -> 再审 -> 收敛`
