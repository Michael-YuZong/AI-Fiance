# Markdown 审稿闭环

## 什么时候触发

只要当前任务会输出一份可独立阅读的研究型 `Markdown` 报告，就默认触发这套闭环。典型包括：

- `stock_pick`
- `briefing`
- `ETF 分析`
- `场外基金分析`
- 其它研究型成稿

并且这条规则默认扩展到以后新增的新功能：

- 不是只有当前列出来的命令才适用
- 只要新功能会对外交付一份 Markdown 成稿，就自动纳入这套闭环
- 不需要用户额外再说一次“也要走外审”

这里的关键判断标准不是“用了哪个命令”，而是“有没有交付一份 Markdown 成稿”。

---

## 默认流程

1. 先生成第一版 `Markdown`
2. 调用外部金融专家审稿器做独立审稿
3. 让外部审稿器对同一个任务独立做一版简洁答案/推荐，并和成稿对比
4. 根据反馈修正代码、数据口径、展示层、交易参数，或吸收独立答案里值得学习的部分
5. 重新生成新版 `Markdown`
6. 对外交付前做一轮发布前一致性校验，确认客户稿和当前详细稿没有漂移
7. 再次审稿
8. 直到外部审稿明确表示没有新的实质问题
9. 最终成稿默认交付“详细解释版”，不是只有结论的“摘要版”
10. 如果用户没有明确要求短版，默认交付完整成稿；自动客户版/摘要版只能作为派生稿，且不能写入 `final`
11. 如果外审意见值得长期采纳，必须沉淀到 [`docs/report_review_lessons.md`](../../docs/report_review_lessons.md) 或对应代码门禁里，不能只修当前这一稿

不要把第一版成稿直接交付给用户。
不要把“只有个股推荐走外审、其它报告先直接交付”当成默认做法。

## 外部审稿器怎么理解

外部审稿器是“第二视角”，默认要做到：

- 脱离项目代码上下文
- 联网核验价格、估值、财报日、新闻、监管、交易参数
- 做最小必要的回测或历史复盘
- 给出明确的反馈意见，而不是泛泛而谈
- 对同一个任务独立做一版简洁答案/推荐，再和成稿对比，提炼值得吸收的地方
- 检查因子完整性：日线稿至少看量价结构/OBV/KDJ/波动压缩；盘中稿至少看 VWAP/开盘缺口/首30分钟；没有竞价/盘口数据时不能写成竞价判断
- 检查催化证据层级：默认应先看财报日历、交易所公告、公司 IR、个股公告，再看新闻补充；`没新闻` 不能直接等于 `没催化`
- 对 A 股个股额外检查：如果 `Tushare` 已有 `披露计划 / 增减持 / 回购 / 分红 / 解禁`，这些结构化事件应优先于新闻标题进入催化和风险判断
- 对 A 股个股额外检查：如果 `Tushare` 已有 `前十大股东 / 前十大流通股东 / 质押统计 / 集合竞价 / 涨跌停边界`，这些结构化数据应优先进入筹码、风险和执行层
- 检查数据降级鲁棒性：如果新闻/事件源降级，会不会把催化和推荐名单打成“假阴性”；成稿是否披露了覆盖率、受影响市场和影响范围
- 检查筹码结构是否真是个股级因子：如果多只股票共用同一条市场级/行业级资金数据，不能把它包装成个股独有优势
- 检查终稿有没有把解释写够；默认成稿必须能回答“为什么这样判断”
- 检查是否存在大量重复模板句；如果不同标的反复复用同一句理由，默认视为解释不合格
- 即使某个任务可联网核验的事实较少，也仍然要保持“脱离当前项目上下文”的独立审稿身份；联网核验不是外审触发门槛

## 哪些任务可以并发

如果用户一次明确要多份互相独立的报告，例如：

- `stock_pick`
- `briefing`
- `ETF 分析`
- `场外基金分析`

可以并发开始生成。  
但要注意两点：

1. 每份报告各自独立产出，不共用“结论稿”
2. 每份报告都要各自完成一轮外部审稿闭环，不能只审其中一份

也就是说，可以并发“开始”，不能并发后“合并糊成一个没审完的交付物”。

---

## 什么时候停止循环

只有在外部审稿明确表示：

- 没有新的 `P0 / P1` 问题
- 剩余主要是展示优化

才可以停止。

如果还有以下问题，就继续循环：

- 事实错误
- 口径混用
- 逻辑断裂
- 展示与评分不一致
- 交易参数硬伤

---

## 这套规则的目的

这套规则不是为了把流程变复杂，而是为了避免两种常见错误：

1. 把“项目内部自洽”误当成“对外可信”
2. 把“第一版 Markdown”误当成“可交付终稿”

它的本质是把工作流固定成：

`生成 -> 外部审稿 -> 修正 -> 一致性校验 -> 再审 -> 收敛`
