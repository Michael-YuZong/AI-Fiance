# AI-Finance 演进路线图

## 1. 这份计划解决什么问题

这不是一份“功能愿望清单”，而是一份把当前项目逐步推进成“更完整 AI 金融专家”的执行计划。

当前项目已经具备：

- 单标研究
- ETF / 场外基金 / 个股推荐
- 风险与组合管理
- 日报与研究问答
- 外审、发布前校验、客户稿导出

但如果按真正金融研究和投资决策的标准看，它还缺：

- 更严格的时点正确性
- 更强的组合构建与风险预算
- 更强的研究问答能力
- 更可信的代理信号与降级披露
- 更完整的事后归因、评分校准和运营闭环

这份计划的目标，是把项目从“高质量投研助手”继续推进到“更接近机构级流程约束的 AI 金融专家”。

---

## 2. 北极星目标

目标不是“功能越来越多”，而是让系统对一个真实投资问题形成完整闭环：

1. 用户提问或发起任务
2. 系统先识别任务类型和对象
3. 拉取当前时点可用的数据与证据
4. 形成研究判断，而不是模块堆砌
5. 把判断映射到组合和风险预算
6. 产出可交付的研究稿或问答结果
7. 经过外部金融专家审稿
8. 修正后再交付
9. 进入后续监控、复盘和经验沉淀

当这条链稳定后，这个项目才算真正接近“AI 金融专家”，而不是多个命令的集合。

---

## 3. 目标状态定义

### 3.1 用户视角

用户希望系统最终能做到：

- 问一个问题，就得到直接答案，而不是拼接式观察
- 研究一个标的，就得到完整证据、风险、执行方案和组合影响
- 做一个推荐，就能说明为什么是它、为什么不是别的
- 数据不完整时，系统会诚实降级，而不是装作确定
- 输出可以直接发给客户或研究同事，而不是先人工大改

### 3.2 金融专业视角

系统最终要满足：

- 时点正确
- 结论可解释
- 交易可执行
- 组合可管理
- 风险可量化
- 结果可复盘
- 规则可持续迭代

### 3.3 工程视角

系统最终要满足：

- 命令行为与渲染合同一致
- 降级路径清楚，不静默吞错
- 外审和发布门禁是正式流程，不是临时动作
- 重要结论都能落到测试、门禁或 lesson

---

## 4. 非目标

这份计划默认不追求下面这些事情：

- 自动交易或无人值守下单
- 高频、超低延迟或盘口级策略系统
- 完全替代人工投资经理
- 一次性做完所有市场、所有资产、所有策略

先把“研究和决策辅助”做到可信，再考虑更自动化的动作。

## 5. 总体工作流

后续所有开发都按下面这个统一流程推进。

### 5.1 单个功能的一轮迭代流程

1. 先定义产品合同
   先说清楚这个功能到底要回答什么问题，输出长什么样，哪些边界必须显式披露。
2. 复现实例
   用真实命令、真实问题或失败测试复现，而不是空想。
3. 修改实现
   优先改真正控制行为的 command / processor / renderer / guard，而不是只改文案。
4. 补测试
   至少补一条能防回归的测试。
5. 生成样例输出
   不是只看测试通过，要看输出是不是像一份真的研究结果。
6. 接入外审
   用外部金融专家 prompt 或研究问答审稿 prompt 再打一轮。
7. 吸收外审意见
   修代码、修口径、修门禁、修渲染、修 tests，不只修当前稿。
8. 更新文档
   至少同步 `AGENTS.md`、必要时补 `docs/report_review_lessons.md` 和这份计划。

### 5.2 一项能力完成的定义

某个能力要被视为“完成”，必须同时满足：

- 有清晰 CLI 或工作流入口
- 输出结构稳定
- 降级逻辑明确
- 有最小必要测试
- 可以接入外审
- 已经和上下游模块打通
- 能说明“什么时候可信，什么时候只作参考”

## 6. 外审闭环怎么接进开发过程

### 6.1 什么时候必须触发外审

以下几类变更默认要触发外审循环：

- 任何会生成正式研究稿的功能
- 会改变推荐名单、推荐顺序、评分或客户结论的逻辑
- 会改变风险、仓位、止损、目标的逻辑
- 会改变问答结构或研究回答方式的逻辑
- 会改变数据降级、覆盖率、证据披露方式的逻辑

### 6.2 用哪个外审 prompt

- 正式 Markdown 报告
  使用 `docs/prompts/external_financial_reviewer.md`
- 研究问答输出
  使用 `docs/prompts/external_research_reviewer.md`
- 报告修订流程说明
  使用 `docs/prompts/report_revision_loop.md`

### 6.3 外审要产出什么

每轮外审至少要产出：

- 总评
- 主要问题
- 可取之处
- 独立答案或独立推荐
- 建议沉淀

### 6.4 什么时候算通过

只有当外审不再提出新的 P0 / P1 问题时，才允许视为这轮开发收敛。

### 6.5 外审意见怎么沉淀

外审意见不能只停在“知道了”。

至少落到下面之一：

- 代码修复
- 测试补齐
- 发布门禁收紧
- 客户稿结构调整
- `docs/report_review_lessons.md`
- `AGENTS.md`
- 本计划的下一轮任务重排

## 7. 模块怎么串成完整 AI 金融专家

最终系统不是并列命令，而是一条串行链路。

### 7.1 任务入口层

负责识别用户到底要什么：

- `assistant`
- `lookup`
- `research`

这层的目标不是“分发命令”，而是识别：

- 问的是标的、组合、市场还是政策
- 是研究问题、交易问题、复盘问题还是日报问题
- 需要即时回答、完整报告，还是进一步跳转其他命令

### 7.2 证据与上下文层

负责把研究需要的背景拼完整：

- 市场 regime
- 宏观与风格背景
- 新闻与结构化事件
- 代理信号
- 组合持仓和 watchlist

这层必须解决“证据够不够”的问题，而不是只负责抓数据。

### 7.3 分析引擎层

负责真正做判断：

- `scan / stock_analysis`
- `compare`
- `discover`
- `fund_pick / etf_pick / stock_pick`
- `policy`
- `risk`

这层的关键是：

- 结论和维度评分一致
- 交易建议和风险一致
- 降级时不会出现系统性假阴性

### 7.4 组合决策层

负责把单标判断转成组合动作：

- `portfolio`
- `risk`
- 后续需要补的组合构建 / 风险预算层

最终它应该回答：

- 能不能买
- 应该买多少
- 买进去后组合会变成什么样
- 这个动作是在放大什么风险

### 7.5 交付层

负责把研究结果变成可用交付物：

- `client_report`
- `opportunity_report`
- `policy_report`
- `release_check`
- `report_guard`

这层必须保证：

- 详细稿和客户稿一致
- 没有过程词泄露
- 没有客户看不懂的内部异常
- 解释性足够，不只是结论

### 7.6 运营与复盘层

负责让系统越用越强：

- `scheduler`
- `decision_review`
- `pick_history`
- 回测与评分校准

这层决定项目能不能从“会生成”进化为“会学习”。

## 8. 分阶段路线图

后续按阶段推进，而不是同时乱改。

## 阶段 A：研究问答升级为真正的研究入口

### 当前推进状态（2026-03-13）

- 已完成核心升级：
  - `research` 已具备市场 / 标的 / 政策 / 组合四类问题分型
  - 市场问答接入了轻量 market diagnosis 和真实 proxy flow，而不是只回退到中性占位
  - 标的问题新增了规则化的场景概率框架，不再只给静态看多/看空语气
  - 证据已经按问题类型排序，而不是简单按模块分组顺序拼接
  - 组合风险问答能给出更具体的集中度、场景和下一步建议
- 阶段 A 现在可以视为“成熟可用”，后续还有优化空间，但不再是最高优先级主战场。

### 目标

把 `research` 从“模块拼装问答”升级成真正的研究入口。

### 要做什么

- 强化问题分型
  - 标的研究
  - 市场诊断
  - 组合问题
  - 政策问题
  - 执行问题
- 建立证据优先级
  不再把所有可用模块都往上贴，而是按问题本身排序证据。
- 强化无标的场景
  “最近市场为什么别扭”这类问题要能回答得像真正研究员。
- 明确代理信号口径
  用了代理信号就明确说是代理。
- 把 `research` 和 `scan / risk / policy / portfolio` 串成统一入口

### 验收标准

- 典型问题能在第一段直接回答
- 证据不再出现明显“模块拼装感”
- 下一步建议是自然延伸，不只是“去跑另一个命令”
- 研究问答外审能连续两轮没有新的 P1
- 有针对性的 command tests 和输出合同测试

### 与其他模块的联动

- 标的问题联到 `scan`
- 组合问题联到 `risk + portfolio`
- 政策问题联到 `policy`
- 市场问题联到 `briefing + regime + proxy signals`

### 外审要求

- 每次改 `research` 结构都跑 `external_research_reviewer`
- 把系统性问题沉淀到 prompt 或 renderer，而不是只改一条回答

---

## 阶段 B：代理信号体系升级

### 目标

让社媒情绪、全球资金流、风格代理等模块从“可用的提示项”变成“有置信度边界的辅助证据”。

### 当前推进状态（2026-03-13）

- 已完成第一步：
  - `social_sentiment` / `global_flow` 增加了显式置信度、限制说明、降级影响字段
  - `research` 已开始消费这套代理合同
  - `briefing` 的资金流说明已明确代理性质和置信度
- 下一步优先做：
  - 把同样的代理披露接进 `stock_pick / fund_pick / etf_pick`
  - 避免市场级代理被写成单标优势
  - 给 renderer 和 release guard 补更严格的代理措辞检查

### 要做什么

- 给代理信号加显式置信度
- 区分硬数据、结构化事件、代理信号
- 在客户稿和研究问答里显式披露代理性质
- 降级时说明影响范围
- 避免把市场级代理错误写成个股级优势

### 验收标准

- 任意输出中，代理信号都不会被表述为已确认事实
- 推荐和问答里能说清楚“这只是辅助证据”
- 外审不再反复指出“代理伪装成硬证据”
- 有 processor tests 和 renderer tests

### 与其他模块的联动

- `research`
- `briefing`
- `stock_pick`
- `fund_pick`
- `etf_pick`

---

## 阶段 C：政策和宏观链路升级

### 目标

把 `policy` 和宏观从“解释器”升级成“能稳定影响研究与推荐的证据层”。

### 当前推进状态（2026-03-13）

- 已完成：
  - `policy` 已能输出模板置信度、匹配别名、政策方向、阶段、时间线提示和 watchlist 映射
  - 官方 URL / 公告页 / 用户提供长正文现在会显式输出 `来源判断 / 抽取覆盖 / 附件标题`
  - 公告页正文事实已经开始按动作密度排序，不再轻易被纯日期标签和公文套话占位
  - `research` 已接入政策问题入口，不再要求先手动切到 `policy`
- 仍待完成：
  - 附件 PDF / OFD 原文的稳定抽取
  - 更严格的政策 taxonomy
  - 宏观和政策在 pick / risk 里的更深联动

### 要做什么

- 改善长文、PDF、公告类正文抽取
- 强化政策分类法
- 把政策影响更明确映射到：
  - 标的
  - 行业
  - ETF / 基金
  - watchlist
  - 组合暴露
- 统一宏观指标在不同模块中的口径
- 让政策与宏观真正进入推荐、问答、晨报和风险链路

### 验收标准

- 对同一政策，不同输出模块给出的方向判断不互相打架
- 长文类政策抽取质量显著提升
- 宏观指标不再只停留在背景描述层
- 有 policy engine tests、output tests、至少一组真实样例回归

### 与其他模块的联动

- `policy`
- `research`
- `briefing`
- `scan`
- `fund_pick / etf_pick / stock_pick`
- `risk`

### 外审要求

- 对政策解读类输出默认走正式金融外审
- 审核重点放在“有没有过度推断”和“是否真的映射到交易层”

---

## 阶段 D：组合构建与风险预算升级

### 目标

把现有 `portfolio + risk` 从“风险报告 + 仓位记录”升级为“组合决策中枢”。

### 当前推进状态（2026-03-13）

- 已完成 v1：
  - `portfolio whatif` 已能回答单笔动作对仓位、波动、Beta、主题/地区暴露的影响
  - pick 报告、`discover`、`briefing` 已接入统一的 `portfolio whatif` handoff
  - `research` 已能直接回答“适合上多少仓位 / 做得进去吗”这类问题
- 仍待完成：
  - 更正式的组合级风险预算和主题上限引擎
  - 推荐到组合动作的自动预算映射

### 要做什么

- 引入组合级风险预算
- 回答单标动作对整体组合的影响
- 加入仓位建议生成逻辑
- 区分：
  - 总仓位
  - 单票上限
  - 单主题上限
  - 分批节奏
- 引入相关性、主题集中度、回撤容忍度约束

### 验收标准

- 对任意一条推荐，系统能回答“买多少更合理”
- 风险报告能映射到实际持仓动作，而不只是统计表
- 压力测试结果能反映到仓位建议
- 至少有一组组合样例从推荐 -> 加仓 -> 风险变化 -> 复盘闭环

### 与其他模块的联动

- `stock_pick / fund_pick / etf_pick`
- `research`
- `portfolio`
- `risk`
- `decision_review`

### 外审要求

- 对组合建议默认检查交易可执行性
- 重点审仓位是否和波动、相关性、事件窗口匹配

---

## 阶段 E：时点正确性与证据溯源升级

### 目标

这是最重要的长期升级之一。
要尽量保证系统用的是“当时看得到的数据”，不是事后知道的数据。

### 当前推进状态（2026-03-13）

- 已完成 v1：
  - `portfolio` 交易日志和 `retrospect` 已保留 `decision_snapshot / execution_snapshot`
  - `research` 已新增独立的“证据时点与来源”段
  - `scan / stock_analysis / stock_pick_detailed / fund_pick / etf_pick` 已能显式展示行情 as_of、催化证据 as_of、催化来源和时点边界
  - 统一 provenance helper 已落地，point-in-time 口径不再散落在各命令里
- 阶段 E 现在可以视为“第一轮收口完成”。
- 后续剩余工作：
  - `policy` 长文原文链路的时点边界
  - 历史回测 / signal confidence 更严格的 point-in-time fixtures

### 要做什么

- 统一时点字段和可见性定义
- 区分：
  - 交易日收盘后可见
  - 次日盘前可见
  - 盘中可见
  - 季度滞后可见
- 对新闻、基金持仓、评级、宏观数据修订、财报披露计划建立更严格的时间边界
- 给关键证据保留来源和时间
- 对历史回测、复盘、样本置信度补强时点说明

### 验收标准

- 外审不再反复指出“事后视角污染”
- 回测和复盘使用的证据口径可解释
- 重要输出能列出关键证据来源与时间
- 有专门的 point-in-time tests 或 fixtures

### 与其他模块的联动

这不是单独一个命令，而是会影响：

- `scan`
- `research`
- `policy`
- `briefing`
- `stock_pick / fund_pick / etf_pick`
- `decision_review`
- `signal_confidence`

## 阶段 F：评分校准、归因与自我学习升级

### 目标

把系统从“会解释”推进到“会验证自己的解释是否真的有效”。

### 当前推进状态（2026-03-13）

- 已完成 v1：
  - `decision_review / retrospect` 已能输出 excess return、setup bucket 校准和结果归因
- 仍待完成：
  - 更系统的评分校准报表
  - 把 lesson 正式写回评分、门禁和 pick 流水线

### 要做什么

- 做评分与真实后验表现的校准
- 做推荐结果归因
  - 是选对了
  - 还是市场整体抬上去
  - 还是止损和仓位救了结果
- 识别哪些维度最常误导
- 把经验写回：
  - 评分
  - 风险
  - 门禁
  - 文案

### 验收标准

- 能回答“高分是否真的更好”
- 能回答“某次推荐成功/失败主要因为什么”
- 有月度或周度归因产物
- 外审意见里不再频繁出现“结论说得好，但没有历史验证”

### 与其他模块的联动

- `pick_history`
- `decision_review`
- `stock_pick / fund_pick / etf_pick`
- `research`
- `signal_confidence`

## 阶段 G：执行成本与可成交性升级

### 目标

把“研究结论”推进到“可执行交易建议”，避免系统只会说该不该买，却回答不了是否做得进去、做进去要付出什么代价。

### 当前推进状态（2026-03-13）

## 9. 跨阶段的硬验收标准

无论做哪一阶段，最终都要过下面这些共同验收。

### 9.1 产品合同验收

- 入口清楚
- 输出结构稳定
- 结论和证据一致
- 明确写出边界和降级

### 9.2 工程验收

- 有针对性测试
- 至少一条真实命令样例可运行
- 不引入静默截断、静默忽略或静默降级

### 9.3 金融验收

- 没有明显事实错误
- 没有明显交易自相矛盾
- 没把代理信号装成硬证据
- 没把事后知道当成当时知道

### 9.4 外审验收

- 走过对应 prompt 的独立审稿
- 无新的 P0 / P1
- 值得长期保留的问题已沉淀到代码、测试或门禁

## 10. 具体推进顺序

后续默认按下面顺序推进，不建议同时大面积改多个阶段。

1. 阶段 C：政策与宏观链路升级
2. 阶段 B：代理信号升级收尾
3. 阶段 H：调度与运营闭环
4. 阶段 F：评分校准、归因与自我学习深化
5. 阶段 G：执行成本与可成交性深化
6. 阶段 D：组合预算深化
7. `discover` v2：把发现链路从规则筛选推进到更像 pick 前置入口的高质量 pre-screen

这个顺序的理由是：

- 阶段 D / E / F / G 的第一轮闭环已经做出来了，当前不再是“从 0 到 1”
- `policy` 和代理信号仍是最明显的证据层短板，会直接影响研究和推荐可信度
- `scheduler` 是最清晰的基础设施短板，应该承接已经成熟起来的内容层
- 后续更值钱的是“深化和统一”，不是继续堆新入口
